# Конфиденциальность данных

## Почему это ядро продукта, а не опция

База приложения – это, по сути, карта для вора: «ПТС в верхнем ящике комода»,
«запасные ключи у соседа», «деньги под матрасом». Утечка такой базы опаснее
утечки переписки. Поэтому конфиденциальность здесь – часть продукта, а не
навесная фича.

## Модель угроз

| Угроза | Статус | Чем закрыто |
|---|---|---|
| Утечка данных по сети | закрыто | нет разрешения INTERNET; всё on-device |
| Другое приложение читает файлы | закрыто | внутреннее хранилище приложения |
| Облачный бэкап уносит базу | закрыто | `allowBackup="false"` + `dataExtractionRules` (cloud-backup) |
| Перенос телефон→телефон (D2D) уносит базу | закрыто | `dataExtractionRules` (device-transfer) |
| Аналитика/крэш-репорты втихую шлют данные | закрыто | нет INTERNET + правило «никаких сетевых SDK» |
| Потеря/кража телефона, копирование файла базы | закрыто | SQLCipher, raw-key из Android Keystore |
| Подсмотрели в разблокированном телефоне | закрыто | биометрия / app-PIN гейт на чтение |
| Свой человек знает системный PIN телефона | закрыто | отдельный код приложения (НЕ системный) на чтение |
| Клавиатура запоминает названия вещей | закрыто | `textNoSuggestions` на полях ввода |
| Форензика с рутованного устройства | частично | шифрование; ключ без обязательной аутентификации (см. решение ниже) |

## Принятые решения

1. **Аутентификация: гейт только на чтение.** Вход в поиск и список – биометрия
   ИЛИ отдельный код приложения. Запись остаётся в одно касание (правило ТЗ «5
   секунд / одно касание»). Логика: утечка – это когда кто-то *читает* карту;
   добавить запись картой не является. Вор с украденным телефоном локаций не увидит.

   **Код приложения – отдельный, НЕ системный.** Это закрывает угрозу «свой
   человек знает PIN телефона» (родственник, тот кто подсмотрел код): системный
   замок его бы не остановил, отдельный код – останавливает. Биометрия берётся
   ТОЛЬКО как `BIOMETRIC_STRONG`, без `DEVICE_CREDENTIAL`, иначе система предложила
   бы телефонный PIN – дыра вернулась бы. Код 4–8 цифр (пожилому проще пароля), его
   держит сын; ежедневно отец входит по биометрии, код – запасной путь. Гейт –
   один раз за запуск процесса.

   **Это UI-гейт, а не крипто-привязка.** Код НЕ входит в ключ шифрования (иначе
   ломалось бы «одно касание» на записи и терялись бы данные при забытом коде).
   Он останавливает человека; против рута/форензики с разблокированным телефоном
   защищает само шифрование базы (Keystore/StrongBox), а не код. Хеш кода (PBKDF2 +
   соль, лок от перебора) короткий PIN делает брутфорсимым офлайн при доступе к
   файлу – но такой доступ требует рута, при котором данные и так читаются через
   приложение; поэтому уровень адекватен модели угроз (инсайдер, а не лаборатория).

2. **Шифрование – всегда, не опция.** База через SQLCipher шифруется всегда.
   Формулировка ТЗ «ключ в Keystore» реализуется корректно: SQLCipher нужен
   пароль-строка, а ключи Keystore неэкспортируемы, поэтому генерируется
   случайный пароль базы, **оборачивается AES-ключом из Android Keystore**,
   шифротекст пароля хранится в файле, при старте разворачивается.

3. **Обёрточный ключ – без `setUserAuthenticationRequired`.** Пароль базы
   разворачивается молча. Это защищает от копирования файла базы и потери
   телефона, не мешая скорости и фоновой записи. Обязательная привязка к
   биометрии дала бы сильнее против форензики, но сломала бы быстрый цикл записи –
   сознательно не берём.

4. **«Недавние» гасим, скриншоты по возможности оставляем.** Экраны чтения
   (список/поиск/правка) убирают содержимое из снимка переключателя задач: на
   Android 13+ через `setRecentsScreenshotEnabled(false)` (обычные скриншоты
   работают – отец сможет прислать скриншот для удалённой помощи, как в исходном
   решении), на Android <13 – через `FLAG_SECURE` (там гранулярности нет, закрыть
   утечку важнее). Прежнее «FLAG_SECURE не ставим» пересмотрено: появился замок
   чтения, а миниатюра в «недавних» его обходила (нашло ревью этапа 5). Ручной
   скриншот экрана данных не опасен для целевой угрозы – к нему всё равно надо
   пройти гейт. `MainActivity` не защищаем (там нет чужих данных).

## Стоящие правила (не нарушать)

- **Никогда не добавлять разрешение INTERNET.** Проверять в APK:
  `aapt2 dump permissions app-debug.apk` – должно быть только RECORD_AUDIO
  (и USE_BIOMETRIC на этапе 5).
- **Никаких сетевых SDK** – ни Firebase, ни Crashlytics, ни аналитики. Даже для
  крэш-репортов. Отсутствие INTERNET это принудительно гарантирует, но правило
  должно быть явным, чтобы никто не добавил «на всякий случай».
- **Шифрование всегда включено** – не делать его настройкой.
- **Поля ввода без подсказок клавиатуры** (`textNoSuggestions`) на экранах
  редактирования, чтобы названия вещей не попали в словарь Gboard/облако Google.

## Что сделано (этап 5)

- **Шифрование всегда:** база через SQLCipher (`net.zetetic:sqlcipher-android`).
  Ключ базы – случайные 32 байта, отдаются SQLCipher как raw-key (`x'<hex>'`, без
  KDF). Ключ обёрнут AES-256-GCM ключом из Android Keystore (StrongBox при наличии),
  шифротекст + IV в приватном файле `filesDir`. Обёрточный ключ без
  `setUserAuthenticationRequired` – переживает смену PIN/отпечатка (данные не теряются).
- **Гейт на чтение:** биометрия (`BIOMETRIC_STRONG`) ИЛИ отдельный код приложения,
  один раз за запуск. Экран `UnlockActivity`; поиск/список не грузят данные до входа.
- **`textNoSuggestions`** на полях ввода – названия вещей не попадают в словарь Gboard.
- Нет INTERNET (проверено в APK); `allowBackup="false"` + `dataExtractionRules`.

## Известный риск: потеря ключа = потеря базы

Ключ обёртки живёт в Keystore/TEE и намеренно не бэкапится в облако (ядро приватности).
Обратная сторона: **factory reset, «Очистить данные», удаление приложения, редкий баг
OEM-прошивки** уничтожают ключ (а первые три – и сам файл базы) → доступ к данным не
восстановить. Данные пересоздаваемы (наговорить заново), это не деньги/пароли.
Код обрабатывает потерю gracefully: `PassphraseUnavailableException` → ошибка на
экране, НЕ краш и НЕ тихое стирание. На будущее (вне объёма этапа 5): локальный
зашифрованный экспорт базы под отдельным паролем, скопированный с телефона.
